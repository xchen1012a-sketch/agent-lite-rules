# 文档更新策略

## 更新频率

- L0/L1：通常不更新 docs。
- L2：必要时更新局部说明或测试记录。
- L3：更新 contract / checklist / current phase。
- L4：更新 project-plan / current / phase。

## 禁止事项

- 禁止为了形式同步所有文档。
- 禁止小改动强行创建阶段文件。
- 禁止把临时想法写成已确认项目事实。
- 禁止把未执行的验证写成通过。

## project-plan 维护条件

`project-plan` 只在范围变化、架构变化、阶段变化时更新。普通 bugfix、文案调整、样式小改、局部测试补充不更新 project-plan。

## docs 归属

真实计划和验收文档必须跟着真实工作边界走。

- 模板 `docs/`：只作为示例模板，不写真实项目状态。
- 子仓 `docs/`：记录本子仓自己的计划、验收、契约、回归清单。
- 父目录 `docs/`：只在跨仓任务时记录总协调计划、跨仓契约说明和跨仓验收证据。

## 写入位置判断

- 单子仓任务：写入该子仓的 `docs/`。
- 跨仓任务：父目录 `docs/` 写总协调计划；各子仓 `docs/` 写各自执行和验收。
- 规则模板任务：只改模板 `docs/` 示例，不写真实项目状态。
- 父目录规则删除后：如果父目录仍承担跨仓协调，父目录 `docs/` 可以保留。

禁止把所有计划都塞进规则目录，也禁止把子仓私有计划写到父目录。
## 计划外文件与计划更新

AI 发现计划外文件时，先输出报告，等待用户确认，不把它自动写进计划。

用户确认纳入后再判断文档动作：

- 只是局部补充：记录在本次交付说明即可。
- 影响当前阶段验收：更新 current phase 或 checklist。
- 影响 API/DB/权限/跨仓协作：更新 contract、checklist 和 phase。
- 改变范围、架构或阶段目标：更新 project-plan / current / phase。

禁止把 AI 自己发现但用户未确认的计划外文件写成已确认范围。
