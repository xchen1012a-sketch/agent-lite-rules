# 上下文加载

## 原则

- 小任务不读大计划。
- 不重新生成已有计划。
- 不全量扫描项目。
- 优先读当前文件、相邻测试、当前阶段入口。
- 只有阶段切换、范围变化、架构变更才读 project-plan。

## 按等级读取

- L0：只读当前工具入口文件（`AGENTS.md` 或 `CLAUDE.md`）和对话上下文。
- L1：读命中文件和必要项目事实。
- L2：读相关 skill、源码和测试。
- L3：读相关 skill、契约、清单、当前阶段。
- L4：读 project-planning skill、current、project-plan 和必要轻规则。

## 读取约束

禁止默认读取全部规则、全部 skills、全部 docs 或全量扫描项目。读取大文件前先搜索命中位置，再分段读取。

## 模块化上下文

- 读代码时按模块边界读取：入口、相邻实现、相邻测试、契约，不跨无关模块扩散。
- 写代码前先判断目标文件属于哪个模块；归属不清时先报告，不直接新建或乱放。
- 多模块任务必须按模块分别收集上下文、分别说明影响。
- 输出报告必须按结论、变更、验证、风险、下一步组织。

## 项目事实优先级

- 如果存在规则目录下的 `project-facts.md`，优先读取它。
- 如果不存在 `project-facts.md`，允许根据同目录的 `project-facts.example.md` 创建，但只能填充 `[auto]` 段；`[manual]` 段保持空白模板。
- 启动或接管任务时，运行 `scripts/refresh-project-facts.ps1` 检查并刷新；脚本会在 `project-facts.md` 缺失或 source hash 变化时只更新 `[auto]` 段。
- 没有脚本或脚本失败时，AI 只能基于可验证来源手动更新 `[auto]` 段，并说明失败原因。
- `[auto]` 段只能记录机器可验证事实：依赖、脚本、目录结构、环境变量 key、Git 状态。
- `[manual]` 段只能由用户或已确认计划维护：项目定位、业务域、模块词表、阶段前缀、当前阶段、安全边界。
- AI 不得猜测、自动补全或覆盖 `[manual]` 段。
- `project-facts.example.md` 只是模板，不代表当前项目事实。

## docs 默认不读

`docs/` 默认不读。只有以下情况才读取：

- L3/L4 任务。
- 用户明确询问计划、契约、验收、回归。
- 当前任务需要更新 contract、checklist、current phase。

## 修复/Review 计划上下文

修 bug、找 bug、测试失败恢复、运行异常排查、代码 review 后继续修复时，按任务等级加载计划上下文：

- L0/L1：通常不读 `docs/plans/`，除非用户明确要求写计划。
- L2 及以上：先读 `docs/plans/README.md`；如果存在 `docs/plans/current.md`，再读 current 指向的阶段文件。
- 没有现成计划时，在 `docs/plans/phases/` 下创建本次修复或 review-fix 计划文件。
- 计划文件先写清阶段，再改代码；不得用最终总结代替前置计划。
- 每阶段完成后，把验证证据写回计划文件或在最终输出中明确说明未写回原因。

## Skill 加载优先级

默认使用 `project-first`：项目内 `.ai-spec/skills/` 优先，用户本地 skills 只在项目 skill 明显不足时补充。

用户可以显式指定：

- `project-first`：项目 skill 优先，本地 skill 仅补充。
- `project-only`：只用项目 skill，不读用户本地 skill。
- `local-first`：用户本地 skill 优先，但不得覆盖项目红线、契约和任务路由。

读取用户本地 skill 前必须说明：项目 skill 缺什么、准备读取哪个本地 skill、读取后如何与项目规则合并。禁止修改用户全局配置或把本地 skill 自动写入项目。
## 计划外文件处理

计划外文件是指不在用户本次明确要求、当前 phase、当前计划、契约或验收范围内的文件。

默认规则：先报告，等用户确认，不直接修改。

必须报告的信息：

- 文件路径。
- 为什么发现它。
- 它和当前任务的关系。
- 不处理会有什么影响。
- 建议动作：忽略、纳入本次任务、生成后续任务、更新计划。

允许例外：

- 用户在当前对话中明确点名该文件。
- 该文件阻塞当前任务验收，且只需最小必要修改。

即使触发例外，也必须在动手前说明原因和影响范围。若计划外文件意味着范围、架构、API、DB、权限、部署或跨仓边界变化，必须停下来等用户确认，并按 L3/L4 处理。
