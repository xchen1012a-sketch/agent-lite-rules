# AI 双工具全栈开发操作手册

> 本手册是给用户看的操作说明，不是默认规则源。AI 只有在用户询问协作流程、全栈流程、Git 提交、工具切换、Handoff 时才读取。

## 一句话理解

Claude Code 和 Codex 都只是执行者，不是项目状态源。稳定协作依赖：

```text
Git 分支 + .ai-rules 轻规则 + skills 能力包 + docs/contracts 契约 + 验证结果 + 必要 Handoff
```

一个 AI 不能继续时，另一个 AI 不接聊天记忆，而是接可读取文件、Git 状态和验证证据。

## 目录角色

```text
workspace/          # 父级工作区，只做协调
├── CLAUDE.md       # Claude Code 入口指针
├── AGENTS.md       # Codex 入口指针
├── backend/        # 后端独立仓库，自己有规则和 docs
└── frontend/       # 前端独立仓库，自己有规则和 docs
```

关键原则：

- 父目录只做全栈协调。
- 如果前后端是独立 Git 仓库，禁止在父目录执行提交。
- 后端 Git 操作只在后端仓库执行。
- 前端 Git 操作只在前端仓库执行。
- 每个子仓应该有自己的 `CLAUDE.md`、`AGENTS.md`、`.ai-rules/` 和 `skills/`。

## AI 分工建议

| 场景 | 推荐工具 |
|---|---|
| 业务方案、架构拆解、接口契约初稿 | Claude Code |
| 代码实现、修 bug、验证、提交整理 | Codex |
| Claude 额度不足 | Codex 根据 Handoff 接手 |
| Codex 遇到复杂业务判断 | Claude Code 根据 Handoff 接手 |

无论哪个 AI 开发，都必须遵守当前项目的 `.ai-rules/README.md`。

## 标准全栈开发流程

### 1. 先读入口，不改代码

```text
这是全栈任务。请先读取相关子仓的 CLAUDE.md / AGENTS.md 和 .ai-rules/README.md。
不要改代码，先分析后端/前端落点、接口契约、风险和验证方式。
```

### 2. 先定接口契约

```text
先进入契约阶段，不要改代码。
请输出接口契约：路径、方法、请求参数、响应字段、错误码、权限、数据库影响、前端页面落点、验证方式。
```

契约建议放在相关项目的 `docs/contracts/`。

### 3. 后端先实现

```text
按已确认契约实现后端第一阶段。
只改 backend/，不改前端。
完成后给出验证结果、未验证内容、风险和回滚方案。
```

### 4. 前端再适配

```text
按后端契约适配前端页面和 API。
只改 frontend/，不改后端。
写 API 前先对齐接口文档。
完成后给出 build/lint/联调结果、未验证内容、风险和回滚方案。
```

### 5. 真实联调

```text
进行真实后端联调，不要使用 mock 作为完成依据。
检查接口字段、权限、页面进入、浏览器 console 和 network。
```

### 6. 分仓提交

全栈任务前后端建议使用同名分支：`feature/<name>`。

父目录只做协调；各子仓分别提交。

### 7. 创建 MR/PR

推荐顺序：后端 MR/PR -> 前端 MR/PR。两个 MR/PR 互相引用，后端契约先稳定，前端后合。

## AI 切换流程

### 当前 AI 停手前

```text
请生成本阶段 Handoff。
写清当前分支、已完成内容、修改文件、接口契约、验证结果、未验证风险、回滚方案和下一步。
不要继续改代码。
```

Handoff 建议写到对应子仓的 `docs/`，不要写到规则模板目录。

### 新 AI 接手时

```text
请先读取入口、.ai-rules/README.md、相关 docs/contracts 和 Handoff。
不要改代码，先执行 git status，确认当前分支、未提交文件、已完成内容、剩余 TODO 和风险。
```

接手后第一步不是写代码，而是确认状态。

## Handoff 必须写清

```text
Task：任务目标
Repo / Branch：仓库和分支
Current State：已完成内容
Contract：接口、请求、响应、错误码、权限
Modified Files：修改文件
Remaining TODO：剩余任务
Verification：已验证 / 未验证
Risks：风险
Rollback：回滚方案
Next AI Instruction：下个 AI 第一件事
```

## 常用指令模板

### 只分析，不改代码

```text
先读取相关 AI 入口和 .ai-rules/README.md，不要改代码。
请分析需求落点、涉及文件、接口契约、风险和验证方式。
```

### 后端开发

```text
只改 backend/。不要改前端。
完成后给出验证、未验证、风险、回滚。
```

### 前端开发

```text
只改 frontend/。不要改后端。
完成后给出 build/lint/联调结果、未验证、风险、回滚。
```

### 全栈开发

```text
这是全栈任务。先定契约，不要直接写代码。
后端和前端按两个独立 git 仓库处理。
父目录只做协调，不允许在父目录做 git 操作。
```

### 提交前检查

```text
请分别检查各子项目的 git status。
确认每个文件的归属仓库，不要在父目录提交。
```

## 高风险动作必须先确认

- 修改登录、认证、权限、数据权限。
- 删除表、删字段、清空数据。
- 操作生产数据库或生产服务器。
- 修改核心框架或共享包。
- 新增生产依赖。
- 批量发送消息或批量写入数据。
- push / 创建 MR/PR / 执行部署。

## 完成定义

一个任务不能只说“完成了”，必须说明：

```text
代码改动完成
契约同步完成
数据库影响说明清楚
验证命令执行并记录结果
未验证内容说明原因和风险
回滚方案明确
前后端 Git 状态清楚
必要时 Handoff 已生成
```
