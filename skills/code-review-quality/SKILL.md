# Code Review Quality

## 目标

用工程视角发现真实风险：行为错误、边界遗漏、复杂度膨胀、安全问题、性能退化和测试缺口。

## 适用场景

- 用户要求 review、检查、把关。
- L2/L3/L4 任务完成前的收尾检查。
- 代码改动较大、跨模块、涉及公共接口。
- AI 生成代码需要质量验收。

## 不适用场景

- 纯产品讨论。
- 只要求格式化或文案修改。
- 没有代码或 diff 可看时做最终评审。

## 最小上下文

1. git diff 或本次改动文件。
2. 相关调用方和测试。
3. 相关契约、schema、配置、错误处理。
4. 验证命令输出。

## 五维检查

1. 正确性：主路径、边界、空值、异常、并发、状态恢复。
2. 可读性：命名、结构、重复、分支复杂度、无用抽象。
3. 架构：职责边界、模块依赖、公共 API、可测试性。
4. 安全：输入校验、权限、密钥、路径、日志、外部数据。
5. 性能：N+1、重复请求、阻塞 IO、无限增长、缓存一致性。

## 工作流

1. 先看 diff 范围，判断是否过大。
2. 按五维检查真实缺陷，优先找会导致行为问题的点。
3. 对每个问题给出文件、位置、风险、建议修法。
4. 区分必须修、建议修、后续优化。
5. 检查测试是否覆盖关键路径和回归点。
6. 如果用户要求 review 后继续修复，或 review 发现必须修的问题并获得用户确认，先创建或更新 `docs/plans/phases/REV-<short-name>.md`，再开始改代码。
7. Review-fix 计划必须包含：发现列表、修复优先级、分阶段修复顺序、每阶段影响文件、验证命令、停止条件、回滚方式。
8. 按计划阶段修复，不把建议优化混入必须修复阶段。
9. 每阶段完成后更新计划状态和验证证据；如未写回，最终输出必须说明原因。
10. 若无问题，说明仍未覆盖的风险。

## 变更大小规则

- 大量无关格式化应拆开。
- 功能、重构、迁移、格式化不要混在一个变更里。
- 超过一个清晰主题时，建议拆分。

## 模块化审查

- 检查新增文件是否落在正确模块；`utils`、`common`、`shared` 需要真实复用理由。
- 检查 route/controller、service、repository、client、worker、component、store、test 是否职责混杂。
- 检查命名是否来自项目词表、业务资源或既有约定；临时名、AI 味命名和重复概念要指出。
- 检查单文件是否膨胀：超过约 800 行、函数超过约 50 行、组件超过约 200 行时要求说明或拆分。
- 检查输出报告是否按结论、变更、验证、风险、下一步拆分；不要把未验证项写成通过。

## 停止条件

- 没有 diff 或无法确认实际变更。
- 当前分支存在大量无关改动，无法归因。
- 需要运行敏感外部服务或破坏性命令才能判断。

## 输出要求

- review 时先列问题，按严重程度排序。
- 每条问题必须可行动。
- 不用大段夸奖；没有问题就直接说没有发现阻塞项。
- review 后进入修复时，必须说明修复计划文件路径、当前阶段和下一阶段。
## 强制执行规则

- 遵守 `.ai-rules/skill-contract.md`。
- 本 skill 的工作流、停止条件和验证方式优先于通用建议。
