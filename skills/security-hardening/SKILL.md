# Security Hardening

## 目标

把不可信输入、敏感数据和外部边界处理清楚。默认最小权限、显式校验、失败安全、日志不泄密。

## 适用场景

- 登录、权限、角色、租户、会话、token。
- 用户输入、文件上传、路径、命令、URL、模板渲染。
- 数据库查询、外部 API、webhook、LLM 输出、后台任务。
- CORS、cookie、环境变量、配置、密钥。

## 不适用场景

- 纯 UI 文案。
- 不触碰输入、权限、数据、外部服务的小样式修改。

## 最小上下文

1. 数据流入口和信任边界。
2. auth/authz 中间件或调用方。
3. DTO/schema、文件路径、外部服务 client。
4. 错误处理、日志、配置和测试。

## 工作流

1. 标出资产：账号、数据、文件、密钥、任务、配额、费用。
2. 标出入口：HTTP、表单、上传、CLI、队列、webhook、LLM。
3. 判断权限：谁能做什么，失败时默认拒绝。
4. 校验输入：类型、长度、范围、枚举、路径白名单、MIME/扩展名。
5. 保护输出：错误不泄露堆栈，日志不记录密钥和敏感原文。
6. 保护执行：不直接执行 LLM 生成命令；FFmpeg 参数结构化。
7. 保护外部服务：超时、重试上限、schema 校验、成本和速率限制。
8. 写最小测试或 smoke 覆盖拒绝路径。

## 必须做到

- 参数化查询，不拼 SQL。
- 文件路径只允许白名单目录内解析后的路径。
- CORS 使用明确 allowlist，不用任意来源。
- 密钥只从安全配置读取，不写入日志、文档、前端包。
- 500 响应不暴露内部堆栈。

## 停止条件

- 新增真实外部服务接入但用户未确认。
- 涉及生产库、真实密钥、真实支付或真实用户数据。
- 需要扩大权限、关闭鉴权、放宽 CORS。
- 需要删除、重置或批量改写数据。

## 验证方式

- 至少验证一个允许路径和一个拒绝路径。
- 检查日志、错误响应和前端包不含密钥。
- 对文件/命令/外部服务变更做 smoke 或手工验证记录。
