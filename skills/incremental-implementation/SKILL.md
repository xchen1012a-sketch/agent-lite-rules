# Incremental Implementation

## 目标

把大任务切成能独立验证的薄切片。每一步都保持项目可运行、可回滚、可解释，避免一次性堆大改动。

## 适用场景

- 一个任务会改多个文件或多个模块。
- 新功能、重构、接口联调、阶段任务。
- 用户只给目标，没有给具体实现步骤。
- 你发现自己准备一次性改很多代码。

## 不适用场景

- L0 问答。
- 单文件小文案、小样式、小配置。
- 用户明确要求只输出方案，不改代码。

## 最小上下文

1. 任务目标和验收标准。
2. 相关入口文件、相邻实现、相邻测试。
3. 可用脚本：check、test、smoke、start、stop。
4. L3/L4 才读取契约、阶段文档和回归清单。

## 工作流

1. 先把目标拆成 2-5 个薄切片，每片必须有可观察结果。
2. 选择最小第一片：优先打通主路径，不先做完整抽象。
3. 每片按顺序执行：改动、局部验证、记录结果。
4. 代码始终保持可运行；未完成能力用明确状态、feature flag 或 mock 边界隔离。
5. 不把顺手发现的问题混进当前切片；除非它阻塞验收。
6. 每片结束后检查 diff，确认没有无关格式化和无关文件。
7. 只有用户明确要求时才提交 Git；提交前再次验证。

## 切片原则

- 一片只交付一个用户可见能力或一个技术边界。
- API、DB、权限、安全变化必须单独成片。
- 先纵向打通，再横向补全。
- 抽象来自重复和复杂度，不来自预感。

## 模块化执行

- 每个切片必须写清目标模块、输入、输出、验收方式和不碰的模块。
- 同一切片内不得同时做功能、重构、格式化和迁移，除非它们不可分割且已说明。
- 新文件必须归入已有模块；没有明确归属时先报告目录建议，不随意新建杂项目录。
- 公共工具只在出现第二个真实调用点后抽出；不要为单点需求提前建通用层。
- 一个文件变成多个职责时，优先按 route/service/repository/component/store/test 等边界拆分。

## 停止条件

- 验收目标不清，无法判断做完。
- 需要不可逆 DB/API/权限变更但用户没确认。
- 当前切片导致基础验证失败，且失败原因不明。
- 继续推进必须重构无关模块。

## 验证方式

- 每片至少跑最小相关验证。
- L2 以上任务至少有测试、构建、类型检查或 smoke 之一。
- 最终说明每片完成了什么、验证了什么、哪些没验证。

## 强制执行规则

- 遵守 `.ai-rules/skill-contract.md`。
- 本 skill 的工作流、停止条件和验证方式优先于通用建议。
