# Incremental Implementation

## 目标

把大任务切成能独立验证的薄切片。每一步都保持项目可运行、可回滚、可解释，避免一次性堆大改动。

## 适用场景

- 一个任务会改多个文件或多个模块。
- 新功能、重构、接口联调、阶段任务。
- 用户只给目标，没有给具体实现步骤。
- 你发现自己准备一次性改很多代码。

## 不适用场景

- L0 问答。
- 单文件小文案、小样式、小配置。
- 用户明确要求只输出方案，不改代码。

## 最小上下文

1. 任务目标和验收标准。
2. 相关入口文件、相邻实现、相邻测试。
3. 可用脚本：check、test、smoke、start、stop。
4. L3/L4 才读取契约、阶段文档和回归清单。

## 工作流

1. 先把目标拆成 2-5 个薄切片，每片必须有可观察结果。
2. 选择最小第一片：优先打通主路径，不先做完整抽象。
3. 每片按顺序执行：改动、局部验证、记录结果。
4. 代码始终保持可运行；未完成能力用明确状态、feature flag 或 mock 边界隔离。
5. 不把顺手发现的问题混进当前切片；除非它阻塞验收。
6. 每片结束后检查 diff，确认没有无关格式化和无关文件。
7. 只有用户明确要求时才提交 Git；提交前再次验证。

## 切片原则

- 一片只交付一个用户可见能力或一个技术边界。
- API、DB、权限、安全变化必须单独成片。
- 先纵向打通，再横向补全。
- 抽象来自重复和复杂度，不来自预感。

## 停止条件

- 验收目标不清，无法判断做完。
- 需要不可逆 DB/API/权限变更但用户没确认。
- 当前切片导致基础验证失败，且失败原因不明。
- 继续推进必须重构无关模块。

## 验证方式

- 每片至少跑最小相关验证。
- L2 以上任务至少有测试、构建、类型检查或 smoke 之一。
- 最终说明每片完成了什么、验证了什么、哪些没验证。

## 常见合理化借口

| 借口 | 实际处理 |
|---|---|
| 顺手一起改更快 | 拆出去，除非阻塞当前验收 |
| 先大改再测试 | 小步改，小步测 |
| 现在抽象一下以后省事 | 先等重复和复杂度出现 |
| 还没完但先接上真实服务 | 真实外部服务接入前必须确认 |
## 强制执行规则

- 一旦本 skill 被任务路由命中，必须按本文件工作流执行；不得只把它当作参考建议。
- 必须先读取本 skill 的最小上下文，再改文件、跑命令或给结论。
- 必须遵守项目级红线：不得写入或修改用户全局 Claude / Codex 配置、全局 rules、全局 skills、全局 hooks、全局 MCP。
- 必须优先使用项目 `.ai-spec/skills/`；读取用户本地 skill 前，必须说明项目 skill 缺口和本地 skill 名称。
- 遇到停止条件必须停下来报告，不得绕过、弱化或自行假设用户已确认。
- 涉及真实外部服务、生产数据、权限、安全、DB、文件删除、Git push、全局配置时，必须先取得用户确认。
- 必须保持代码和输出模块化；不得把职责、计划、解释、验证和风险混在一起。
- 最终输出必须说明：使用了哪个 skill、改了什么、验证了什么、未验证什么、是否触发停止条件。
