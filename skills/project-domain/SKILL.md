# Project Domain

## 目标

把通用规则落到当前项目，避免 AI 每次重新猜项目背景、模块边界和命名方式。

## 适用场景

- 任务涉及项目业务域、模块命名、阶段命名、目录归属。
- 通用 skill 能力够用，但需要结合当前项目语义执行。
- 新增模块、移动文件、拆分职责、设计 API/DB/UI 边界。
- 修 bug 或 review 时需要判断影响范围属于哪个项目模块。

## 不适用场景

- 纯 L0 问答，且不需要项目事实。
- 单文件小文案或格式修正。

## 最小上下文

1. `.ai-rules/project-facts.md`；如果缺失，先运行 `scripts/refresh-project-facts.ps1` 生成 `[auto]` 段。
2. `docs/rules/business-rules.md`，如果任务涉及业务规则、领域术语或规则冲突且该文件存在。
3. `docs/plans/current.md`，如果存在。
4. 当前任务涉及的 phase、contract、review 或 checklist。
5. 目标模块入口和相邻测试。

## 工作流

1. 先读取 `project-facts.md`，区分 `[auto]` 和 `[manual]`。
2. 只信任 `[auto]` 中可验证事实；`[manual]` 中 `已确认` 的条目可作为项目语义，`待确认` 只能作为候选。
3. 如果 `[manual]` 缺少项目名称、业务域、模块词表或阶段前缀，AI 可以按明确来源补草案并标 `待确认`，不得静默当作已确认。
4. 如存在 `docs/rules/business-rules.md`，只把有来源的 `confirmed` 规则当作执行依据；`proposed` 规则必须先确认。
5. 根据模块词表判断文件归属；归属不清时，先给候选路径和理由。
6. 把当前任务映射到项目模块、当前阶段和验证边界。
7. 输出时说明本次采用的项目事实来源。

## 项目专属 Skill 生成原则

当一个项目反复出现同类领域任务时，可以新增项目专属 skill，例如：

- 检索、排序、匹配质量。
- 外部服务 provider 与核心业务边界。
- 文档解析、媒体处理、导入流水线。
- 订单、库存、财务、权限、审批等业务域。

新增项目前必须满足：

- 名称来自 `project-facts.md` 中已确认条目、项目计划或用户确认。
- 只固化已确认事实，不把猜测写成规则。
- 和通用 skill 互补，不复制通用 skill 内容。
- 包含适用场景、最小上下文、工作流、停止条件和验证方式。

## 停止条件

- 缺少项目事实，且无法运行刷新脚本。
- `[manual]` 为空且找不到明确来源，任务又需要业务域、模块词表或阶段前缀。
- 用户要求的命名或目录会破坏已有模块边界。
- 需要把猜测写入项目规则、计划或 skill。

## 验证方式

- 检查新增/修改文件是否落在正确项目模块。
- 检查命名是否来自项目事实、已有代码或已确认计划。
- 检查最终输出是否说明项目事实来源。
- L2 及以上任务必须说明读取了哪些项目 rules / skills。

## 强制执行规则

- 遵守 `.ai-rules/skill-contract.md`。
- 本 skill 的工作流、停止条件和验证方式优先于通用建议。
