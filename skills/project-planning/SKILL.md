# Project Planning

## 目标

把范围大或状态不清的工作拆成可确认、可执行、可验证、可恢复的阶段。只在范围、阶段、架构或多仓协作需要时维护计划。

## 适用场景

- 新项目启动、旧项目接入、规则迁移。
- 阶段切换、里程碑验收、跨仓协作。
- 架构、API、DB、权限、进程边界变化。
- 用户明确要求做计划、拆阶段、推进 current。

## 不适用场景

- L0 问答。
- L1 文案、样式、单文件小改。
- 没有范围变化的 bugfix。
- 只补一条已有阶段的验收证据。

## 最小上下文

1. `AGENTS.md` 或 `CLAUDE.md`。
2. `.ai-rules/project-facts.md`，如果存在。
3. `docs/plans/current.md`，如果存在。
4. current 指向的 phase 文件。
5. 只有范围、架构、阶段变化时才读 `project-plan.md`。

## 工作流

1. 先判断是否真的需要计划；小任务不造文档噪音。
2. 未确认计划前，只输出计划草案，不改代码、不建项目、不安装依赖。
3. 用户确认并说“执行”后，先写入计划文件，再开始第一阶段。
4. 已有 `current.md` 时先恢复当前阶段，不重建计划；除非用户确认范围变化。
5. 任务拆成多个可验收阶段，但每次只执行 current 指向的一个阶段。
6. 阶段完成必须有验证证据；未验证不得写成通过，不推进 current。

## 三种模式

### 新项目

- 只根据用户想法输出计划草案。
- 草案必须确认项目名、业务域、模块词表、仓库/子仓命名、阶段前缀。
- 用户确认前不创建文件、不初始化 Git、不安装依赖、不写代码。
- 用户确认后生成 `project-plan.md`、`current.md`、`phases/*.md`。
- Git 只允许初始化在确认后的项目目录；多子仓时父目录不建 Git。

### 老项目接入

- 先只读盘点目录、技术栈、已有规则、docs/plans、scripts、Git 状态和关键入口。
- 输出接管报告，等待用户确认。
- 确认前不生成计划、不覆盖入口、不改业务代码、不初始化 Git、不安装依赖。
- 确认后优先沿用已有计划；没有计划再生成最小计划集。
- 同名冲突生成 `.proposed`，不直接覆盖。

### 阶段推进

- 先读 `current.md` 和当前 phase。
- 执行前报告目标、范围、不做事项、验收标准和计划修改文件。
- 只做当前阶段；计划外文件先报告，等用户确认。
- 阶段完成后记录验证证据，再更新 current。

## 必需产物

- `docs/plans/project-plan.md`：总目标、范围、不做、仓库边界、阶段列表、总体验收。
- `docs/plans/current.md`：只指向一个当前阶段。
- `docs/plans/phases/<PHASE-ID>-<name>.md`：目标、范围、不做、验收证据、回滚方式。

模板见 `docs/plans/project-plan.example.md`、`current.example.md` 和 `docs/plans/README.md`。

## 停止条件

- 项目目标、仓库边界、交付范围或验收标准不清。
- 单仓/多仓、技术栈、数据库或部署形态需要用户确认。
- 旧项目存在规则冲突或未提交改动，继续接入可能覆盖约定。
- 用户未确认计划，却要求进入实际改造。
- 总计划和当前阶段计划缺失，却要开始代码实现。

## 验证方式

- 当前任务确实需要计划。
- 已恢复或创建 current。
- 项目名、模块词表、阶段前缀来自用户确认、项目事实或现有约定。
- 阶段编号唯一，且有验收证据。
- project-plan 只在范围、架构或阶段变化时更新。
- 未执行验证没有写成通过。

## 输出要求

计划阶段：输出总计划草案、阶段列表、每阶段验收标准、需要用户确认的问题。

执行阶段：输出当前阶段、计划更新原因、下一步、阻塞项、验收证据和未验证项。

## 强制执行规则

- 遵守 `.ai-rules/skill-contract.md`。
- 本 skill 的工作流、停止条件和验证方式优先于通用建议。
