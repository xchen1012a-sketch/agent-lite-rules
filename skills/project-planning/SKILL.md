# Project Planning

## 目标

把不清楚、范围大的工作拆成小的、可验证、可恢复的阶段。计划不是仪式；只有当范围、阶段、架构或多人/多仓协作真的需要它时才维护。

## 适用场景

- 新项目启动、旧项目接入、规则迁移。
- 阶段切换、里程碑验收、跨仓协作。
- 架构调整、API/DB/权限/进程边界变化。
- 用户明确要求“做计划”“拆阶段”“推进 current”。

## 不适用场景

- L0 问答。
- L1 文案、样式、单文件小改。
- 没有范围变化的 bugfix。
- 已有阶段计划且本次只补一条验收证据。

## 最小上下文

1. `AGENTS.md`。
2. `docs/plans/current.md`，如果存在。
3. current 指向的 phase 文件。
4. 只有范围、架构、阶段变化时才读 project-plan。

## 工作流

1. 先判断是否真的需要计划。
2. 如果用户还没有确认计划，只输出计划草案，不改代码、不建项目、不安装依赖。
3. 用户确认并说“执行”后，先建立总计划 `project-plan.md` 和多个分阶段执行计划，再进入实现。
4. 如果已有 `current.md`，先恢复当前阶段，不重建计划；除非用户确认范围变化。
5. 把任务拆成多个可验收阶段，但只执行 current 指向的当前阶段。
6. 阶段编号必须全局唯一，例如 `FE-01`、`BE-01`、`DB-01`、`OPS-01`。
7. 每个阶段写清目标、范围、不做事项、验收证据、回滚方式。
8. 只有范围或架构变化才更新 project-plan。
9. 阶段完成后再推进 current，不能凭感觉推进。

## 计划模式

适用于新项目、旧项目接入、架构调整、多子仓协作、阶段切换和用户明确要求“先做计划”的任务。

必须分两步：

1. 计划确认：输出总计划草案和阶段拆分，等待用户确认。
2. 执行准备：用户确认并说“执行”后，先写入总计划和分阶段执行计划，再开始第一阶段。

### 新项目模式

适用于空目录、无既有业务代码、用户只描述“想做什么”的场景。

流程：

1. 只根据用户想法和相关 skills 输出计划草案。
2. 计划草案必须包含项目名称、产品/业务域、模块词表、仓库/子仓命名、阶段前缀。
3. 不创建文件、不初始化 Git、不安装依赖、不写代码。
4. 用户确认计划并说“执行”后，先生成总计划和分阶段执行计划。
5. 若当前目录没有 `.git`，可以初始化项目级 Git 仓库。
6. 再接入 `.ai-spec` 根入口，创建必要目录，进入 current 指向的第一阶段。

必须注意：

- 初始化 Git 只允许发生在当前项目目录。
- 如果计划采用前后端分离多子仓，父级目录不初始化 Git；只在 `frontend/`、`backend/` 等子仓目录初始化 Git。
- 多子仓接入时，父目录可以临时拉取 `.ai-spec` 用于分发；分发完成后，父目录只保留 `CLAUDE.md` / `AGENTS.md` 路由入口，完整 `.ai-spec` 必须落在各子仓。
- 子仓必须各自拥有 `CLAUDE.md` / `AGENTS.md`，并指向本子仓 `.ai-spec`；执行规则不得依赖父目录完整规则源。
- 不创建 remote，不 push。
- 不写用户全局 Claude / Codex 配置。
- 不同时推进多个阶段。
- 不允许 AI 随意发明文件名、模块名、仓库名；命名必须来自项目名、模块词表和已确认计划。

### 老项目模式

适用于已有代码、已有 docs、已有 Git、已有规则或不确定项目状态的场景。

流程：

1. 先只读盘点：目录结构、技术栈、已有规则、已有 docs、脚本、Git 状态、关键入口文件。
2. 输出老项目接管报告：项目事实、已有规则冲突、单仓/多仓判断、建议接入位置、风险、需要用户确认的问题。
3. 用户确认前，不生成计划文件、不覆盖入口文件、不改业务代码、不初始化 Git、不安装依赖。
4. 用户确认后，优先沿用已有计划；没有计划再生成 `project-plan.md`、`current.md` 和 `phases/*.md`。
5. 已有同名规则或计划文件时生成 `.proposed`，等待用户确认替换。
6. 用户说“执行当前阶段”后，只执行 `current.md` 指向的阶段。

必须生成或维护：

- `docs/plans/project-plan.md`：总计划，记录目标、范围、技术路线、仓库边界、项目目录草图、阶段列表、总体验收标准。
- `docs/plans/current.md`：当前阶段入口，只指向一个正在执行的阶段。
- `docs/plans/phases/<PHASE-ID>-<name>.md`：分阶段执行计划，每个阶段独立验收。

禁止：

- 用户未确认计划前直接写代码。
- 只有总计划，没有阶段执行计划。
- 只有阶段列表，没有验收证据。
- 同时执行多个阶段。
- 阶段未验收就推进 current。
- 把计划外文件自动纳入计划。

## 阶段模板

```md
# BE-01 API 基线

- 状态：in-progress
- 更新时间：YYYY-MM-DD

## 目标
- 建立后端 API 基线。

## 范围
- health / ready
- 错误响应 envelope
- API smoke

## 不做
- 不接生产数据库
- 不接真实外部 AI 服务

## 验收证据
- [ ] health smoke 通过
- [ ] 单测通过
- [ ] 契约已更新

## 回滚
- 移除本阶段新增 API 和配置。
```

## 常见合理化借口

| 借口 | 实际处理 |
|---|---|
| 小改也顺手补计划 | 不补，避免文档噪音 |
| current 不存在，直接重建全套计划 | 先建最小 current，不虚构长期路线 |
| 阶段完成了但没跑验证 | 不推进阶段 |
| 计划越细越安全 | 计划只细到下一步可执行 |

## 红旗

- 没读 current 就改 project-plan。
- 小任务创建一堆 phase 文件。
- 阶段没有验收证据。
- project-plan 里写了未经确认的事实。
- 阶段编号重复或只用 `01/02/03`。

## 验证清单

- [ ] 当前任务确实需要计划。
- [ ] 已恢复或创建 current。
- [ ] 项目名、模块词表、阶段前缀已确认。
- [ ] 总计划包含项目目录草图，且目录命名来自项目名、模块词表或现有约定。
- [ ] 阶段编号唯一。
- [ ] 阶段有验收证据。
- [ ] 未执行验证没有写成通过。
- [ ] project-plan 只在范围/架构/阶段变化时更新。

## 输出要求

计划阶段输出：总计划草案、阶段列表、每阶段验收标准、需要用户确认的问题。

执行阶段输出：当前阶段、是否更新计划、更新原因、下一步、阻塞项、验收证据和未验证项。
## 停止条件

- 项目目标、仓库边界、交付范围或验收标准不清。
- 需要决定单仓/多仓、技术栈、数据库或部署形态但用户未确认。
- 旧项目存在未提交改动或规则冲突，继续接入可能覆盖现有约定。
- 用户还没有确认计划书，不进入实际改造。
- 总计划和分阶段执行计划没有建立，不进入代码实现。
## 强制执行规则

- 一旦本 skill 被任务路由命中，必须按本文件工作流执行；不得只把它当作参考建议。
- 必须先读取本 skill 的最小上下文，再改文件、跑命令或给结论。
- 必须遵守项目级红线：不得写入或修改用户全局 Claude / Codex 配置、全局 rules、全局 skills、全局 hooks、全局 MCP。
- 必须优先使用项目 `.ai-spec/skills/`；读取用户本地 skill 前，必须说明项目 skill 缺口和本地 skill 名称。
- 遇到停止条件必须停下来报告，不得绕过、弱化或自行假设用户已确认。
- 涉及真实外部服务、生产数据、权限、安全、DB、文件删除、Git push、全局配置时，必须先取得用户确认。
- 必须保持代码和输出模块化；不得把职责、计划、解释、验证和风险混在一起。
- 最终输出必须说明：使用了哪个 skill、改了什么、验证了什么、未验证什么、是否触发停止条件。
