# Debugging Recovery

## 目标

遇到失败时先停下来定位事实，不靠猜测叠补丁。用复现、缩小范围、根因修复和回归验证恢复系统。

## 适用场景

- 测试、构建、lint、typecheck 失败。
- 服务启动失败、端口冲突、进程闪退。
- 页面空白、交互失效、接口异常、数据不一致。
- 修一个 bug 后出现另一个 bug。

## 不适用场景

- 已经有明确根因和明确修复的小改。
- 纯需求讨论。
- 用户明确只要解释错误信息。

## 最小上下文

1. 完整错误输出或可复现现象。
2. 最近改动 diff。
3. 相关日志、network、console、数据库状态或测试用例。
4. 最小运行/测试命令。

## 工作流

1. 停止继续加功能，先保存当前证据。
2. 对 L2 及以上调试/修复任务，先创建或更新 `docs/plans/phases/FIX-<short-name>.md`；如果已有 `docs/plans/current.md`，同步指向该修复计划。
3. 计划文件必须写清：现象、复现方式、初始证据、影响范围、不做事项、2-5 个修复阶段、每阶段验证、停止条件、回滚方式。
4. 复现：确认失败可重复，记录命令、URL、输入、时间点。
5. 分层定位：配置、启动、路由、DTO、业务、DB、外部依赖、前端状态、浏览器。
6. 缩小范围：找最小失败用例或最小请求。
7. 找根因：解释为什么失败，不只描述哪里失败。
8. 修复：按计划阶段改最少必要代码，避免扩大范围。
9. 防回归：能写测试就写测试，不能写测试就补 smoke 或明确手工验证。
10. 复验：重跑失败用例，再跑相邻验证。
11. 每阶段结束后更新计划状态和验证证据；如未写回，最终输出必须说明原因。

## 停止条件

- 无法复现，且继续修改只能靠猜。
- 需要连接真实生产/远程敏感环境。
- 日志或错误包含密钥、隐私、令牌，需要先脱敏。
- 修复需要破坏数据、重置仓库或清空状态。

## 验证方式

- 必须说明原失败如何复现。
- 必须说明根因和修复点。
- 必须说明复验命令或手工步骤。
- L2 及以上必须说明修复计划文件路径和当前阶段状态。
- 未能复验时说明具体阻塞原因。

## 证据边界

- `logs/` 只用于观察运行事实；不要把日志文件当成状态源修业务逻辑。
- `fixtures/`、`repro/` 保存最小复现输入；不得混入真实敏感数据。
- 修复代码只落在根因模块；测试或 smoke 放在相邻测试边界。
- 临时调试输出、console、print、debug endpoint 必须在收尾时移除或转为结构化日志。
- 如果根因跨 API/DB/UI/部署边界，先列影响模块，再分别调用对应 skill。

- 遵守 `.ai-rules/skill-contract.md`。
- 本 skill 的工作流、停止条件和验证方式优先于通用建议。
