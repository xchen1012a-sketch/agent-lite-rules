# Debugging Recovery

## 目标

遇到失败时先停下来定位事实，不靠猜测叠补丁。用复现、缩小范围、根因修复和回归验证恢复系统。

## 适用场景

- 测试、构建、lint、typecheck 失败。
- 服务启动失败、端口冲突、进程闪退。
- 页面空白、交互失效、接口异常、数据不一致。
- 修一个 bug 后出现另一个 bug。

## 不适用场景

- 已经有明确根因和明确修复的小改。
- 纯需求讨论。
- 用户明确只要解释错误信息。

## 最小上下文

1. 完整错误输出或可复现现象。
2. 最近改动 diff。
3. 相关日志、network、console、数据库状态或测试用例。
4. 最小运行/测试命令。

## 工作流

1. 停止继续加功能，先保存当前证据。
2. 复现：确认失败可重复，记录命令、URL、输入、时间点。
3. 分层定位：配置、启动、路由、DTO、业务、DB、外部依赖、前端状态、浏览器。
4. 缩小范围：找最小失败用例或最小请求。
5. 找根因：解释为什么失败，不只描述哪里失败。
6. 修复：改最少必要代码，避免扩大范围。
7. 防回归：能写测试就写测试，不能写测试就补 smoke 或明确手工验证。
8. 复验：重跑失败用例，再跑相邻验证。

## 停止条件

- 无法复现，且继续修改只能靠猜。
- 需要连接真实生产/远程敏感环境。
- 日志或错误包含密钥、隐私、令牌，需要先脱敏。
- 修复需要破坏数据、重置仓库或清空状态。

## 验证方式

- 必须说明原失败如何复现。
- 必须说明根因和修复点。
- 必须说明复验命令或手工步骤。
- 未能复验时说明具体阻塞原因。

## 常见合理化借口

| 借口 | 实际处理 |
|---|---|
| 可能是缓存问题 | 证明缓存相关，再清理 |
| 环境问题先忽略 | 记录证据，缩小到环境边界 |
| 改这里应该能好 | 先建立因果链 |
| 失败不稳定不用管 | 间歇失败也要收集触发条件 |
## 强制执行规则

- 一旦本 skill 被任务路由命中，必须按本文件工作流执行；不得只把它当作参考建议。
- 必须先读取本 skill 的最小上下文，再改文件、跑命令或给结论。
- 必须遵守项目级红线：不得写入或修改用户全局 Claude / Codex 配置、全局 rules、全局 skills、全局 hooks、全局 MCP。
- 必须优先使用项目 `.ai-spec/skills/`；读取用户本地 skill 前，必须说明项目 skill 缺口和本地 skill 名称。
- 遇到停止条件必须停下来报告，不得绕过、弱化或自行假设用户已确认。
- 涉及真实外部服务、生产数据、权限、安全、DB、文件删除、Git push、全局配置时，必须先取得用户确认。
- 最终输出必须说明：使用了哪个 skill、改了什么、验证了什么、未验证什么、是否触发停止条件。
