# Testing Verification

## 目标

用证据结束任务，而不是用“看起来没问题”结束任务。测试失败时先复现、定位、缩小范围，再修复和加防护。

## 适用场景

- 任何行为变更。
- bugfix。
- API / DB / 进程 / 外部服务相关改动。
- 前端交互、浏览器行为、构建或部署验证。
- 用户要求验收、回归、检查质量。

## 不适用场景

- 纯问答。
- 纯文档措辞修改且不影响行为。
- 用户明确只要草稿，不要求验证。

## 最小上下文

1. 本次变更文件。
2. 相邻测试。
3. 项目测试脚本：package、pyproject、Makefile、scripts。
4. contract/checklist/current phase，只有 L3/L4 需要。

## 工作流

1. 先判断风险等级：纯函数、组件、API、DB、进程、外部服务。
2. 选择最小相关验证，不一开始跑最大全集。
3. bugfix 先写或找到能复现 bug 的测试。
4. 测试失败时按顺序处理：复现、定位、最小化、修复、加防护。
5. 涉及 API/DB/进程时跑 smoke。
6. 涉及浏览器时检查 console、network、DOM 或截图。
7. 没跑的测试必须说明原因，不能写成通过。
8. 同一个成功命令不要无意义重复跑；代码变了再重跑。

## 验证层级

- unit：纯逻辑、工具函数、状态转换。
- integration：API、数据库、文件、多个模块边界。
- e2e：关键用户路径。
- typecheck：类型和 DTO。
- lint：风格和静态问题。
- build：打包和编译。
- smoke：服务启动、health、核心 API、主页面。
- manual verification：必须记录具体步骤和观察结果。

## 文件边界

- `tests/unit/` 覆盖纯逻辑、状态转换和小函数；不得依赖真实网络、真实 DB 或真实外部服务。
- `tests/integration/` 覆盖 API、repository、文件、mock 外部服务和模块边界。
- `tests/e2e/` 覆盖关键用户路径；用用户动作命名，不按组件内部实现命名。
- `fixtures/`、`testdata/` 只放测试输入和样例数据；不得混入真实密钥、真实用户数据或生产导出。
- `mocks/`、`fakes/` 封装外部服务替身；mock 行为要和 contract 对齐。
- `scripts/test.*` 是测试入口；不要把一次性长命令只留在聊天记录里。
- 回归测试应贴近 bug 所在模块，不为一个 bug 新建无归属的大测试文件。

## 选择指南

| 改动类型 | 最小验证 |
|---|---|
| 文案/样式 | 构建或截图检查 |
| 纯函数 | 单测 |
| 组件交互 | 组件测试或浏览器检查 |
| API | API 单测 + smoke |
| DB migration | migration 执行 + repository/API 测试 |
| 进程/部署 | health + smoke |
| bugfix | 复现测试 + 回归测试 |

## 常见合理化借口

| 借口 | 实际处理 |
|---|---|
| 改动很小不用测 | 做最小相关验证 |
| 先修，测试以后补 | bugfix 先复现再修 |
| 构建通过就代表功能好 | 构建只证明能编译，不证明行为 |
| 测试失败可能是环境问题 | 记录证据，先定位，不随便忽略 |
| 没跑但我读了代码 | 静态阅读不能写成通过 |

## 红旗

- 最终报告没有验证命令。
- 使用“应该”“看起来”代替结果。
- 测试失败后随机改代码。
- API 改了但没有 smoke。
- DB 改了但没有 migration 验证。
- 浏览器 UI 改了但没看实际渲染。

## 验证清单

- [ ] 已选择与风险匹配的最小验证。
- [ ] 行为变更有测试或明确无法测试原因。
- [ ] 失败先定位再修。
- [ ] L3 改动有 smoke 或解释原因。
- [ ] 未执行验证单独列出。
- [ ] 最终结论基于真实输出。

## 输出要求

输出验证表：命令、目的、结果、关键输出摘要。另列未验证项、原因、风险和下一步。
## 停止条件

- 需要真实生产数据、真实密钥、真实外部服务才能验证。
- 测试会破坏数据、清空状态、发送真实消息或产生费用。
- 失败无法复现，继续修改只能靠猜。
- 当前验证命令不明确，无法区分通过、跳过和未执行。
## 强制执行规则

- 一旦本 skill 被任务路由命中，必须按本文件工作流执行；不得只把它当作参考建议。
- 必须先读取本 skill 的最小上下文，再改文件、跑命令或给结论。
- 必须遵守项目级红线：不得写入或修改用户全局 Claude / Codex 配置、全局 rules、全局 skills、全局 hooks、全局 MCP。
- 必须优先使用项目 `.ai-spec/skills/`；读取用户本地 skill 前，必须说明项目 skill 缺口和本地 skill 名称。
- 遇到停止条件必须停下来报告，不得绕过、弱化或自行假设用户已确认。
- 涉及真实外部服务、生产数据、权限、安全、DB、文件删除、Git push、全局配置时，必须先取得用户确认。
- 必须保持代码和输出模块化；不得把职责、计划、解释、验证和风险混在一起。
- 最终输出必须说明：使用了哪个 skill、改了什么、验证了什么、未验证什么、是否触发停止条件。
