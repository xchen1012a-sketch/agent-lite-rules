# Frontend Web

## 目标

做真实可用的用户界面，而不是空壳页面或模板感 UI。页面要能完成目标流程，状态完整，响应式稳定，可访问，视觉遵守项目已有设计系统。

## 适用场景

- 新建或修改页面、组件、布局。
- 接 API、状态管理、表单、列表、筛选、上传、画布或工作台。
- 修复交互、视觉、响应式、可访问性问题。

## 不适用场景

- 纯后端 API。
- 纯数据库迁移。
- 不涉及用户界面的脚本或配置任务。

## 最小上下文

1. 目标页面或组件。
2. 相邻组件、样式、主题 token。
3. API client、mock 或 contract。
4. 相关测试和现有启动/构建命令。

## 工作流

1. 明确用户要完成的真实流程，不先做空 landing。
2. 查已有设计系统、组件库、图标库、颜色和间距规则。
3. 分离数据容器和展示组件；API 调用集中到 client/service。
4. 为 loading、error、empty、success、disabled 状态写 UI。
5. 移动端优先，至少考虑 320、768、1024、1440 宽度。
6. 文字不能溢出，按钮和控件不能互相遮挡。
7. 交互元素必须可键盘访问，有可理解 label。
8. 不使用泛滥的紫色渐变、大圆角卡片、堆卡片、假 hero 等 AI 默认审美，除非项目设计系统就是这样。
9. 完成后跑最小相关验证；涉及浏览器行为时做截图或 Playwright/DevTools 检查。

## 组件边界

- 页面负责布局和数据入口。
- 容器负责请求、加载、错误和状态转换。
- 展示组件只负责渲染和用户动作回调。
- 表单校验靠 schema 或统一规则，不散落在模板里。
- URL state 用于筛选、分页、可分享状态；不要滥用全局 store。

## 文件边界

- `pages/`、`routes/` 只组织路由级布局、数据入口和页面状态，不沉淀可复用业务组件。
- `components/` 只放可复用 UI；单个组件超过约 200 行或混入请求逻辑时必须拆分。
- `features/<domain>/` 放领域组件、hooks/composables、局部 store 和领域类型。
- `api/`、`services/`、`clients/` 集中封装请求、错误映射和 contract 适配；组件不得直接散落 `fetch/axios`。
- `stores/` 只放跨页面状态；局部状态保留在页面或 feature 内。
- `types/` 只放共享类型；API 类型优先来自契约或生成文件，不手写猜字段。
- `styles/`、`tokens/` 放主题、变量和全局样式；不得在组件里堆重复魔法数。
- 测试文件贴近被测模块；E2E 按用户流程命名，不按实现细节命名。

## 相邻 Skill 触发

- 需要新增或改变 API 字段、错误形状、权限或分页语义时，同时触发 `backend-api`。
- 需要端到端流程、真实浏览器验证或回归矩阵时，同时触发 `testing-verification`。
- 涉及上传、导出、音视频、图片处理或素材库时，同时触发 `media-pipeline`。

## 验证清单

- [ ] typecheck 通过或说明无法执行。
- [ ] build 通过或说明无法执行。
- [ ] 单测/组件测试覆盖关键逻辑。
- [ ] loading/error/empty 状态可见。
- [ ] 关键断点无溢出。
- [ ] 浏览器无明显 console error。
- [ ] API 字段来自契约或已确认来源。

## 输出要求

说明修改页面/组件、用户流程、状态覆盖、验证命令、截图/浏览器检查结果、未验证项和剩余风险。
## 停止条件

- 需要改变全局设计系统、路由结构或状态模型但用户未确认。
- API 字段、权限或错误形状不清，继续会靠猜契约。
- 需要接入真实支付、登录、上传、媒体或外部服务。
- 响应式布局、核心交互或可访问性无法完成最小验证。
