# Database

## 目标

让数据模型可演进、可恢复、可审计。数据库变更必须先设计，再迁移；高风险操作先停下来确认。

## 适用场景

- 新表、新字段、新索引、新约束。
- migration、数据修复、软删除、审计字段。
- PostgreSQL / MySQL / SQLite 通用设计。
- Redis、缓存和长期业务事实的边界判断。

## 不适用场景

- 纯 UI 改动。
- 不涉及持久化的后端逻辑。
- 临时本地测试数据清理。

## 最小上下文

1. 当前 schema / migrations。
2. 相关 model/entity。
3. repository/query。
4. 业务规则和 API contract。
5. 数据库测试或迁移命令。

## 工作流

1. 先写设计：实体、关系、约束、查询模式、增长预期。
2. 判断数据是否需要长期保存；长期事实进数据库，不进 Redis。
3. migration 只新增，不修改已上线历史迁移。
4. 新字段优先考虑默认值、可空性、回填策略和兼容窗口。
5. 索引服务于真实查询；写清为什么需要这个索引。
6. 明确软删除、审计字段、外键和级联策略。
7. 生产数据、删除表、批量更新、破坏性 DDL 必须暂停确认。
8. 执行迁移前先确认环境，不默认连接真实生产库。
9. 迁移后跑相关测试或最小查询验证。

## 设计检查点

- 主键策略清楚。
- 唯一性约束清楚。
- 读写路径清楚。
- 数据生命周期清楚。
- 回滚或恢复策略清楚。

## 模块边界

- `migrations/` 只放结构或数据迁移；不得写业务查询、API 逻辑或运行时分支。
- `schema/`、`models/`、`entities/` 描述表结构、关系、枚举和约束；不得直接调用外部服务。
- `repositories/`、`dao/`、`queries/` 只封装持久化访问；不得混入 HTTP、权限、媒体或 LLM 编排。
- `services/` 承担业务事务和规则编排；不得散落 raw SQL，除非项目已有约定且有测试覆盖。
- `seed/`、`fixtures/` 只放开发或测试数据；不得伪装成正式迁移。
- `scripts/` 中的数据修复必须支持 dry-run、范围过滤和执行报告。
- 表名、字段名、migration 文件名必须来自项目词表或已确认模型，不为方便随意起名。

## 相邻 Skill 触发

- 改 API 可见字段、DTO 或错误语义时，同时触发 `backend-api`。
- 改页面依赖字段或筛选/分页语义时，同时触发 `frontend-web`。
- 改生产风险、权限、审计、脱敏或密钥相关字段时，同时触发 `security-hardening`。

## 停止条件

以下情况必须暂停询问用户：

- 删除表、删除字段、truncate。
- 批量修改生产数据。
- 修改历史 migration。
- 连接真实生产数据库。
- 迁移不可逆且没有备份/回滚说明。

## 验证清单

- [ ] migration 可执行。
- [ ] 没有修改历史 migration。
- [ ] 相关 model/repository 已同步。
- [ ] API/DTO 影响已说明。
- [ ] 索引和约束有理由。
- [ ] 高风险操作已获得确认。
- [ ] 回滚或恢复策略明确。
