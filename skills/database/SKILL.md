# Database

## 目标

让数据模型可演进、可恢复、可审计。数据库变更必须先设计，再迁移；高风险操作先停下来确认。

## 适用场景

- 新表、新字段、新索引、新约束。
- migration、数据修复、软删除、审计字段。
- PostgreSQL / MySQL / SQLite 通用设计。
- Redis、缓存和长期业务事实的边界判断。

## 不适用场景

- 纯 UI 改动。
- 不涉及持久化的后端逻辑。
- 临时本地测试数据清理。

## 最小上下文

1. 当前 schema / migrations。
2. 相关 model/entity。
3. repository/query。
4. 业务规则和 API contract。
5. 数据库测试或迁移命令。

## 工作流

1. 先写设计：实体、关系、约束、查询模式、增长预期。
2. 判断数据是否需要长期保存；长期事实进数据库，不进 Redis。
3. migration 只新增，不修改已上线历史迁移。
4. 新字段优先考虑默认值、可空性、回填策略和兼容窗口。
5. 索引服务于真实查询；写清为什么需要这个索引。
6. 明确软删除、审计字段、外键和级联策略。
7. 生产数据、删除表、批量更新、破坏性 DDL 必须暂停确认。
8. 执行迁移前先确认环境，不默认连接真实生产库。
9. 迁移后跑相关测试或最小查询验证。

## 设计检查点

- 主键策略清楚。
- 唯一性约束清楚。
- 读写路径清楚。
- 数据生命周期清楚。
- 回滚或恢复策略清楚。

## 停止条件

以下情况必须暂停询问用户：

- 删除表、删除字段、truncate。
- 批量修改生产数据。
- 修改历史 migration。
- 连接真实生产数据库。
- 迁移不可逆且没有备份/回滚说明。

## 常见合理化借口

| 借口 | 实际处理 |
|---|---|
| 先改旧 migration 最快 | 不改历史，新建 migration |
| 数据量现在小，不用索引设计 | 按查询模式判断，不按当前数据量猜 |
| Redis 先存长期状态也行 | Redis 只做缓存/短期状态，不做事实源 |
| 字段没人用可以直接删 | 先确认消费者和回滚路径 |

## 红旗

- migration 和 model 不一致。
- 新字段不可空但没有默认值或回填。
- 查询新增过滤/排序但没有索引判断。
- 把生产库当成本地库。
- 数据修复脚本没有 dry-run 或备份策略。

## 验证清单

- [ ] migration 可执行。
- [ ] 没有修改历史 migration。
- [ ] 相关 model/repository 已同步。
- [ ] API/DTO 影响已说明。
- [ ] 索引和约束有理由。
- [ ] 高风险操作已获得确认。
- [ ] 回滚或恢复策略明确。
